# 假装手写.skill

假装手写.skill 用来从用户过去写过的源码里蒸馏个人编码习惯，并把这些习惯转成后续可执行的写码规则。它不是通用代码规范生成器，而是“个人风格建模器”。它的最终交付物不应只是一份报告，而应是一份完整的“模仿用户代码风格的 Skill 包”。

## 它解决什么问题

- 用户给你一批仓库、目录、文件，想让你总结“我平时怎么写代码”
- 用户希望你继续补功能时尽量像他本人写的
- 用户有多门语言，希望区分跨语言共性和语言专属习惯
- 用户不想要泛泛而谈的“命名要清晰、函数要短小”这类空话，而是要能落地的模仿规则
- 用户希望在分析后，直接拿到一份可复用的模仿 Skill，用于后续持续按本人风格写代码

## 核心设计

这个 skill 参考了 女娲.skill 的结构化蒸馏思路，但对象从“人物思维与表达方式”切换成“程序员编码习惯”。核心变化有三点：

1. 调研对象变成用户代码语料，而不是公开人物材料。
2. 提炼对象变成命名、架构、模式、注释、包划分、测试等编码习惯。
3. 必须做语言隔离，避免把某种语言的惯性错误迁移到另一种语言。
4. 需要下钻到函数内部，分析局部变量命名和语法偏好，而不只停留在类和目录层。
5. 需要把命名语言质量纳入画像，例如英文熟练度、错词复现、拼音或中文命名混入。
6. 需要识别 AI 生成或强 AI 辅助代码，避免把这类样本误判成作者本人的稳定手写风格。
7. 需要估计作者的稳定编码水平和能力边界，并把“本人会写”和“更像靠 AI 才完成”的部分区分开。
8. 需要识别没有署名的协作代码，避免把合作方风格误并入用户本人的编码画像。
9. 对多人项目要先建立本人基线，再做来源归因和作者簇建模，避免直接把整个项目平均成一个人。
10. 需要区分“用户本人稳定手写”“用户本人加 AI 强辅助”“其他人类协作者”“模板生成物”。
11. 识别协作者时要看项目中心性，优先找核心骨架作者，而不是先被实验和调试文件带偏。

## 使用方式

典型触发语句：

- 按我的风格写这个 Python 模块
- 根据这个仓库总结我的编码习惯
- 看看我写 Java 和 Python 时有哪些不同
- 模仿我现有代码，把这个 service 补完

## 输出应该包含什么

- 样本范围和可靠性判断
- 跨语言共性
- 各语言的高置信度习惯、弱信号、禁止误迁移项
- 可以直接用于后续代写的模仿规则摘要
- 可直接交给另一个 AI 使用的模仿 Skill 成品
- 命名语言质量画像，例如标准英文、错拼英文、拼音、中文直译、多语混用
- 函数内部细粒度画像，例如局部变量命名、switch 与 if-else 偏好、早返回与嵌套风格
- AI 介入样本判断，以及这些样本是否被排除或降权
- 可疑协作样本判断，以及这些样本是否被排除或待确认
- 作者簇建模，包括核心协作者簇和边缘协作者簇
- 来源分类，包括本人稳定手写、本人加 AI 强辅助、其他协作者、模板生成物
- 作者稳定能力边界判断，避免把 AI 补强结果误当作者常态水平
