# 模仿 Skill 模板与片段模式

此文件定义 Phase 5 需要产出的“模仿用户代码风格的 Skill 包”模板。目标不是导出一个独立 JSON，而是生成一个标准 Skill 目录，供另一个模型直接加载和使用。

## 标准产物结构

```text
generated-style-skill/
├── SKILL.md
├── README.md
├── references/
│   ├── style-profile.md
│   └── representative-snippets.md
└── evals/
    └── evals.json
```

## 各文件职责

### 1. SKILL.md

必须包含：

- `name` 与 `description`
- 适用场景与触发语句
- 执行顺序：先读取风格报告，再读取代表性片段，再按规则生成代码
- 仿写决策顺序：生成时先看近邻文件，再看目标语言规则，再看跨语言共性，最后才做保守补全
- 证据优先级：style-profile 的高置信度特征优先于抽象概括，representative-snippets 的局部微习惯优先于模型自发“优化”
- 跨语言共性规则
- 分语言模仿规则
- 运行时约束：禁止重新做来源归因，禁止依赖完整源码或历史项目标签做判断
- 保守补全策略

## 推荐工作流

1. 先完成样本覆盖与风格分析，整理出高置信度规则、弱信号、来源边界和禁止误迁移。
2. 再整理代表性代码片段，优先覆盖不同语言、不同层级、不同来源类别。
3. 最后再生成完整模仿 Skill 包，不要在分析早期就开始拼装下游 Skill。

关键约束：

- 下游模仿 Skill 只能消费上游已经整理好的结论，不能把“识别这是不是某个历史项目、目录、协作者簇、AI 区域”当作运行时任务。
- `禁止误迁移` 与 `来源边界` 留在上游风格报告中即可，不要求作为下游生成 Skill 的固定章节出现。
- 如果一条边界必须依赖完整源码背景才能判断，就不要把它写进下游 Skill 的执行规则。

建议骨架：

```markdown
---
name: imitate-user-style
description: |
  根据已整理好的用户代码风格报告与代表性代码片段，按用户本人风格继续写代码。
---

### 2. README.md

必须说明：

- 这个 Skill 的目标是什么
- 依赖哪些输入样本整理而来
- 使用时应提供什么上下文
- 这个 Skill 不解决什么问题

### 3. references/style-profile.md

这是完整代码风格报告。结构应至少包含：

- 样本范围
- 特征总表：把所有发现的特征列出来，每条写明特征、归因对象、信号强度、置信度、证据、纳入策略
- 频率概览：注释频率、常用内容频率、各自的信号强度、置信度和证据
- 跨语言共性
- 来源分类
- 协作者簇
- 分语言画像
- 能力边界
- 诚实边界

### 4. references/representative-snippets.md

这是给下游模型直接模仿的代表性片段集合。每个片段至少记录：

- 语言
- 路径
- 用途说明
- 建议上下文行数
- 片段正文

采集时优先覆盖：

- 不同语言
- 不同模块层级（入口、骨架、业务实现、测试、脚本）
- 不同来源类别（本人稳定手写、本人加 AI、协作者、模板/生成产物中的对照片段）
- 不同写法类型（命名、控制流、错误处理、对象组织、API 调用）

示例格式：

```markdown
## snippet-python-01
- 语言：Python
- 路径：src/app/service.py
- 用途：请求参数校验与转换
- 建议上下文：4 行

```python
def parse_user_input(payload):
    try:
        data = json.loads(payload)
        return data
    except ValueError:
        return {}
```
```

### 5. evals/evals.json

至少包含 3 类测试：

- 给定现有样本后能否产出正确的风格报告
- 给定风格报告后能否按目标语言模仿写代码
- 面对可疑 AI / 协作者样本时能否维持来源边界

## 片段采集要求

- 优先选高置信度、能体现命名和控制流习惯的片段
- 不要只选“看起来高级”的片段，要选最能代表作者稳定微习惯的片段
- 优先保留原始命名和局部注释
- 有敏感信息时做最小脱敏，不要破坏风格信号
- 尽量让片段覆盖高频写法、边缘写法和可疑写法的对照，以便后续区分用户偏好与外部 API / 框架 / 语言常规
- 片段也要服务于频率判断：保留足够多能说明注释频率和常用内容频率的样本，而不是只采最精致的一两个片段

## 输出要求

- 最终交付物默认是完整 Skill 包
- 只有在用户明确拒绝时，才退化为仅输出分析报告
- 不要把 JSON、YAML 或规则列表误当成最终 Skill 成品
