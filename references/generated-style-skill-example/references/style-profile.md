# 示例风格报告

## 样本范围

- 语言：Python、TypeScript
- 核心样本：订单服务、状态解析、测试用例、少量脚本
- 本人基线样本：低污染个人项目中的核心业务代码与主要测试
- 排除样本：模板代码、自动生成文件、可疑协作者目录、注释密度异常高的包装层
- 可靠性判断：中高；核心业务与测试覆盖较好，脚手架与包装层已降权
- AI 介入样本：兼容层、转换层、解释性注释异常密集的包装代码
- 本人加 AI 样本：局部包装代码与少数完整度异常高的适配层
- 可疑协作样本：注册骨架中的另一套稳定命名与控制流风格

## 特征总表

- 近邻文件优先、少做无根据重构：归因对象：用户偏好；信号强度：强；置信度：高；证据：核心业务与测试均复现；纳入策略：纳入
- 早返回多于深层嵌套：归因对象：用户偏好；信号强度：强；置信度：高；证据：Python 业务函数与 TypeScript 状态分支；纳入策略：纳入
- 语义化局部变量名多于 `tmp` / `data` 泛名：归因对象：用户偏好；信号强度：中；置信度：高；证据：订单服务、测试构造段；纳入策略：纳入
- named export：归因对象：外部 API / 项目约定；信号强度：中；置信度：中；证据：TypeScript 服务层稳定复现，但更像项目组织约定；纳入策略：降权
- 注释密度在业务层偏低、在包装层偏高：归因对象：混合信号（用户偏好 + AI/包装层影响）；信号强度：强；置信度：中；证据：核心业务与包装层对比；纳入策略：业务层保留低注释倾向，包装层仅作对照
- 错误处理偏保守兜底：归因对象：用户偏好；信号强度：中；置信度：中；证据：状态解析与输入转换逻辑；纳入策略：纳入

## 频率概览

- 注释频率：Python 与 TypeScript 核心业务层注释较少，测试中只保留必要说明，包装/兼容层注释密度明显更高；信号强度：强；置信度：中；证据：业务文件与包装层对比
- 常用内容频率：高频出现订单、状态、解析、汇总等职责词；常见模式是“输入校验 -> 早返回 -> 小范围转换 -> 保守兜底”；信号强度：强；置信度：高；证据：核心业务、测试场景与服务层复现

## 跨语言共性

- 偏好保持近邻文件风格，不轻易大改结构。信号强度：强；置信度：高；证据：Python 与 TypeScript 样本都维持原有分层与调用顺序
- 复杂逻辑更重可读性，不热衷过早抽象。信号强度：中；置信度：高；证据：业务函数与状态解析分支
- 注释更偏“必要时补充说明”，而不是高密度解释每一行实现。信号强度：中；置信度：中；证据：业务层与包装层对比

## 写法归因

- named export：这是项目约定或 TypeScript 常规写法；信号强度：中；置信度：中；证据：服务层稳定出现，但缺乏与 default export 的充分对照；模仿策略：仅在近邻文件也采用时保留
- 早返回：更像用户个人偏好；信号强度：强；置信度：高；证据：多语言复现且超出单一框架要求；模仿策略：保留
- 低注释业务层：更像用户偏好；信号强度：中；置信度：中；证据：核心业务中稳定出现；模仿策略：保留，不主动补大量解释性注释

## 来源分类

- 用户本人稳定手写：核心业务层、主要测试场景用例
- 用户本人 + AI 强辅助：兼容层、转换层、注释异常密集的包装代码
- 其他人类协作者：注册骨架中的另一套稳定命名与控制流风格
- 模板 / 自动生成 / 第三方：脚手架初始化文件与生成代码

## 协作者簇

- 主要职责：模块挂载与统一入口包装
- 风格特征：注释更密、命名更规整、控制流更模板化

- 核心性判断：中
- AI 介入判断：中

## Python 画像

### 高置信度习惯
- 命名：倾向 snake_case 和语义化局部变量名。信号强度：强；置信度：高；证据：订单服务、测试构造函数；模仿规则：优先使用语义化短中长度命名
- 控制流：偏好早返回，避免过深嵌套。信号强度：强；置信度：高；证据：输入校验、汇总函数；模仿规则：优先 guard clause

### 命名语言质量
- 观察：以标准英文与业务语义词为主，少量缩写但基本稳定。信号强度：中；置信度：高；证据：`summary_rows`、`order_items` 等；模仿策略：保留

### 函数内部细粒度偏好
- 局部变量：偏语义名，如 `summary_rows`
- 控制流：输入为空时先返回，主路径保持短
- 临时对象/中间状态：允许使用清晰的中间列表，不刻意压成一行

### 代表性代码片段
- 业务层、测试层和少量工具层都应覆盖，优先展示早返回、局部语义命名和低注释风格

### 频率特征
- 注释频率：业务层低、测试中低到中；信号强度：中；置信度：中；证据：核心业务与测试文件
- 常用内容频率：订单、汇总、解析、状态等语义词高频；信号强度：强；置信度：高；证据：服务层与测试构造段；模仿规则：延续业务语义命名

### 弱信号
- 类型标注密度中等偏上，但样本不足以证明对所有模块都适用。信号强度：弱；置信度：中；证据：少量服务函数；模仿策略：仅参考

## TypeScript 画像

### 高置信度习惯
- 导出与组织：维持既有 named export 风格。信号强度：中；置信度：中；证据：服务层；模仿规则：优先跟随近邻文件
- 状态处理：更偏查表或显式分支，而不是抽象过深的模式层。信号强度：强；置信度：高；证据：状态解析函数；模仿规则：先写清楚分支，再考虑抽象

### 命名语言质量
- 观察：标准英文较稳定，状态枚举词清晰。信号强度：中；置信度：高；证据：`resolveOrderState`、`pending`、`paid`；模仿策略：保留

### 函数内部细粒度偏好
- 局部变量：偏短而清晰
- 控制流：顺序分支明显，多用直接返回
- 临时对象/中间状态：能不用就不用

### 代表性代码片段
- 服务层与测试层都应覆盖，重点展示显式分支、命名和保守兜底

### 频率特征
- 注释频率：服务层低，包装层高；信号强度：强；置信度：中；证据：服务函数与包装层对比；模仿规则：默认保持低注释密度
- 常用内容频率：状态解析、映射、兜底返回高频；信号强度：强；置信度：高；证据：状态函数与周边逻辑；模仿规则：优先显式处理常见状态

### 弱信号
- 更复杂的状态管理模式样本不足。信号强度：弱；置信度：低；证据：缺少大型组件样本；模仿策略：仅参考

## 模仿规则摘要
1. 优先服从近邻文件与目标模块语境，再应用抽象规则。
2. 业务层保持低到中等注释密度，不主动生成高密度解释性注释。
3. 延续高频语义词、早返回和保守兜底的组织方式。

## 详细总结
- 已发现的特征需要分成用户偏好、项目约定、标准写法、AI/协作者影响几个桶来理解，不能混写。
- 注释频率和常用内容频率是关键复制信号：前者帮助判断“像不像人写”，后者帮助判断“像不像这个人写”。

## 能力边界
- 作者稳定会写：中小型业务逻辑、状态解析、输入转换、测试场景覆盖
- 疑似需 AI 强辅助：包装层、大段解释性注释、完整度异常高的兼容代码
- 疑似来自他人协作：注册骨架与部分初始化适配层
- 判断依据：注释密度、控制流模板化程度、命名与抽象断裂

## 诚实边界
- 缺少更大型前端状态管理与复杂异步流样本
- TypeScript 导出风格可能受项目约定影响，不能完全上升为个人审美
- 注释频率在包装层受 AI / 模板污染，不能直接外推到所有代码
