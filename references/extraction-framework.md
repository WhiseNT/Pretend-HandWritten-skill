# 编码习惯蒸馏方法论

本文件定义“什么算用户习惯，什么不算”。主 skill 在正式提炼前应先读取本文件。

## 一、证据优先级

### 0. 覆盖优先原则

在正式提炼之前，先确保样本覆盖足够广，再谈“代表性”。

不要因为少数文件看起来更整洁、更复杂或更核心，就默认它们足以代表作者整体风格。少量精选阅读容易带来三类偏差：

- 只读核心模块，忽略作者在测试、脚本、胶水层里的稳定微习惯
- 只读高质量文件，把本人加 AI 或核心协作者风格误当成作者常态
- 只读近邻高相似文件，错过同一作者在不同层级下的稳定变化

最低覆盖要求：

1. 每种目标语言至少覆盖 3 类样本中的 2 类以上：核心业务、辅助/胶水、测试/脚本。
2. 多人项目至少覆盖 3 个层级中的 2 个以上：入口/骨架、业务实现、实验/包装。
3. 任何高置信度结论都不能只来自单个目录、单个文件簇或单一职责层。

如果做不到以上覆盖，不要假装“已经足够全面”；应把结论降级，并明确指出偏差来源。

从高到低排序：

1. 用户明确指定的核心业务代码
2. 同一仓库中反复出现的手写模块
3. 用户的测试代码和工具脚本
4. 配置和胶水代码
5. 自动生成文件、脚手架初始代码、依赖产物

如果样本来自多人项目，还要额外排序：

1. 用户本人低污染项目或目录
2. 用户明确说明“这是我自己写的”的样本
3. 与本人基线高度连续的核心业务代码
4. 多人项目中的来源不明样本
5. 自动生成文件、第三方模板、缓存和构建产物

第 5 类通常不用于提炼个人风格，除非用户明确说明这是自己长期维护过并深度修改的代码。

## 二、三重验证

只有同时通过下面至少两项，才值得进入“高置信度习惯”：

### 1. 复现性

同类写法在多个样本中稳定出现，例如：

- 同一类命名在 3 个以上文件出现
- 同一分层策略跨 2 个以上模块出现
- 同一种注释方式同时出现在业务代码和测试代码中

反例：

- 只在一个设计得很完整的核心模块里出现
- 只在同一目录下的相邻文件里出现
- 只在一个疑似 AI 强辅助区域里出现

这些都不应直接升级为高置信度习惯。

### 2. 特异性

该习惯能把这个用户与一般“写得还算规范的人”区分开。比如：

- 喜欢以 action + object 命名 service，而不是泛泛的 manager
- 喜欢把 schema、service、repository 明确分目录，而不是按页面聚合
- 几乎不写块注释，但会给复杂 SQL 或边界条件写解释性注释

### 3. 可执行性

该结论必须能改写成具体写码规则，例如：

- “Python 中倾向在边界层使用 pydantic，在内部层保留原生 dataclass”
- “Java 中 controller 很薄，业务编排放在 service，DTO 转换显式写在 assembler”

如果只能写成“代码比较整洁”“注释比较清晰”，说明它不可执行，应降级或删除。

## 三、重点观察维度

### 1. 命名层

观察：

- 类、函数、变量、常量命名模式
- 包 / 模块 / 文件命名模式
- 是否偏好领域词、动宾词、缩写、语义前缀
- 测试命名是行为式、场景式还是 should 风格
- 命名语言是否稳定使用英文，还是混入拼音、中文直译、错拼英文、项目内部黑话

#### 命名语言质量子维度

额外检查：

- 英文是否基本准确，还是出现稳定的错拼但作者持续沿用
- 是否在顶层名词用英文、局部变量退回拼音/中文音译
- 是否大量使用 data、obj、tmp、test、handler 这类泛词来回避更精确的英语
- 是否用中文拼音表达业务概念，例如 baoShi、faZhen、yaoShui 这类模式
- 是否出现“中文思维直译英文”而不是社区常用词

### 2. 结构层

观察：

- 目录如何拆：按层、按领域、按功能、按页面、按包
- 是否存在稳定的入口层 / 领域层 / 基础设施层
- 是否习惯抽 adapter、assembler、helper、util、base

### 3. 抽象层

观察：

- 喜欢先抽接口，还是先写具体实现
- 喜欢小函数拼装，还是在单个方法中顺序展开
- 是否偏好 declarative 风格
- 是否乐于提前抽象，还是偏保守

### 3.5 函数内部层

观察：

- 局部变量名是完整语义词、缩写、单字母还是临时泛名
- 参数命名是否和调用场景强绑定
- 是否习惯创建 result/data/map/obj 这类中间对象
- 是否偏好先声明一堆变量再逐步赋值，还是边算边返回
- 分支偏好：switch、if-else 链、查表、早返回、continue、break、嵌套层级
- 循环偏好：for、forEach、map/filter/reduce、iterator、while

### 4. 注释层

观察：

- 注释密度高还是低
- 更常写“做什么”“为什么这么做”“边界条件”“TODO/NOTE/FIXME”哪一种
- 文档字符串是否规范，是否会写参数、返回值、异常说明
- 注释语言是英文、中文、双语，还是命名与注释语言不一致

### 5. 约束层

观察：

- 错误处理是否显式
- 是否喜欢提早返回
- 是否保留 assert / guard clause
- 日志是否在边界层集中打印

### 6. 测试层

观察：

- 测试文件位置和命名
- 用例组织方式：按函数、按类、按场景、按 Given/When/Then
- mock 的使用频率和风格

## 四、应当降权的情况

### 多人项目且缺少本人基线

如果一个项目明显多人协作，但还没有建立用户本人基线，不要直接从该项目提炼“用户稳定风格”。

此时更合理的输出是：

- 风格分簇
- 来源不确定性
- 建议补充哪些低污染样本

### 框架默认代码

例如：

- Spring Boot 初始化类
- Django/Vite/Next 默认模板
- ORM 自动生成 model

这些代码主要反映框架，不反映用户。

### 格式化器强制结果

如果某些差异明显只是 formatter、linter 或 IDE 自动导入造成，不要把它当个人品味。例如：

- import 排序
- 行尾逗号
- 花括号换行
- 统一 quote 风格

除非用户在局部手工打破默认规则且稳定复现。

### 团队规范污染

若仓库明显是多人维护、命名极其统一、注释模板固定，先判断这是不是团队规范而非个人风格。此时应更多观察：

- 用户手写的业务决策点
- 局部 helper / utility / adapter 的真实写法
- commit 较新的核心模块

### AI 生成或强 AI 辅助污染

如果某段代码同时出现以下多个特征，应降权处理，而不是直接纳入用户稳定习惯：

- 与同仓库其余文件相比，命名、注释、分层和异常处理突然异常规整
- 教科书式解释和边界覆盖非常完整，但近邻文件普遍没有这种密度
- 大量使用看似正确但缺少项目语境锚点的抽象名词
- 单个模块的实现复杂度显著超过作者其余手写样本
- 风格像“一次性整段生成”，而不是逐步演进出来

遇到这种情况：

1. 先标为“可疑 AI 样本”
2. 默认降权或仅作弱信号
3. 如果该样本会显著影响最终画像，向用户确认是否纳入

### 用户本人加 AI 强辅助

如果某段代码与用户主风格存在断裂，但仍保留明显的用户语义习惯、项目语境锚点或领域偏好，同时又带有较强 AI 或 Vibe Coding 痕迹，应优先考虑它属于“用户本人加 AI 强辅助”，而不是直接归到其他协作者。

常见信号：

- 模块主题和领域词仍紧贴用户其他项目
- 注释、兼容性说明、fallback 说明异常密集，但命名或局部实现仍保留用户习惯
- 代码更像包装层、说明层、转换层、DSL 或兼容层，而不是另一位作者长期维护的核心框架骨架

遇到这种情况：

1. 先标为“本人加 AI 样本”
2. 不并入“稳定基线画像”
3. 可单独作为“扩展风格”或“能力补强来源”输出

### 无署名协作污染

如果某段代码同时出现以下多个特征，应考虑它可能来自其他开发者，而不是用户本人：

- 命名、注释、抽象层次、控制流和文件组织同时与用户主样本断裂
- 这种断裂呈现为另一套稳定的人类写法，而不是 AI 常见的过满解释或过度标准化
- 该风格在某个子目录或一组文件中持续复现，像是另一位作者的局部地盘
- 与该模块相邻的用户样本无法形成自然演进关系

遇到这种情况：

1. 先标为“可疑协作样本”
2. 默认不纳入高置信度用户画像
3. 如果没有明确署名，直接向用户确认是否为合作方所写

### 核心协作者优先级

在多人项目中，优先沿以下位置寻找核心协作者，而不是先盯实验和调试区域：

1. 入口与注册点
2. 抽象骨架和核心接口
3. provider / registry / meta / model
4. 依赖求解 / 调度 / 排序
5. pack / resource / adapter / path / zip / mod 这类统一分发层
6. 业务实现层
7. 实验 / 调试 / 测试层

占据越靠前的位置，越可能代表项目核心协作者，而不是边缘作者。

## 五、输出分级规则

### 高置信度习惯

满足：证据强、可复现、能指导写码。

### 弱信号

满足：有倾向，但样本少、只出现在单一目录，或可能受框架影响。

### 禁止误迁移

满足：只在某语言、某框架、某目录下成立，不能推广到其他语境。

### 可疑 AI 样本

满足：风格断裂明显，但尚未得到用户确认。默认不并入高置信度画像。

### 本人加 AI 样本

满足：代码更像用户主导下的 AI / Vibe Coding 产物。默认不并入稳定基线，但可作为扩展风格单独输出。

### 可疑协作样本

满足：风格断裂明显，且更像另一位作者的稳定写法，但尚未得到用户确认。默认不并入高置信度画像。

### 协作者簇

满足：一组文件在职责、抽象层次、命名、注释、错误处理和建模方式上高度一致，并共同承担项目骨架的一部分。

## 六、如何从观察转成规则

不要停留在观察句，要写成可执行约束。

示例：

- 观察：函数名多以 load_/build_/resolve_ 开头，极少使用 generic process/do/handle
- 规则：继续写同类逻辑时，优先使用具象动词 + 领域对象命名，避免空泛动词

- 观察：复杂逻辑很少用长注释解释流程，但会在边界校验和兼容性分支上补原因注释
- 规则：新增代码保持低注释密度，只在异常路径、兼容分支和性能权衡处补原因说明

- 观察：局部变量常用 result、data、tmp，且很少使用具象名词
- 规则：继续模仿时保留这种偏泛的临时变量命名，不要突然全面换成学术化精确命名

- 观察：作者在复杂枚举分支里偏好 switch，很少写长 if-else 链
- 规则：处理离散状态时优先使用 switch；不要随意改成对象映射或多层 if-else

- 观察：命名中稳定混用拼音与英文，例如核心术语长期用拼音表示
- 规则：对该领域词保留作者原有拼音/混合命名，不要擅自英文正规化到失去作者辨识度

- 观察：某模块的抽象、注释和错误处理质量远高于其余样本，且风格明显断裂
- 规则：先把该模块标成可疑 AI 强辅助样本，不纳入作者高置信度画像，必要时向用户确认来源

- 观察：作者在中低复杂度功能中稳定表现一致，但只有极少数高复杂度模块显得异常完整
- 规则：把“常规能力边界”和“疑似 AI 补强能力边界”分开写，后续模仿优先依赖前者

- 观察：某一组文件长期呈现另一套命名和控制流习惯，且不像同一作者的自然波动
- 规则：先把这组文件标为可疑协作样本，不纳入用户高置信度画像，并向用户确认是否为他人编写

- 观察：某组文件更像用户本人在 AI 强辅助下完成，风格断裂集中在注释密度、包装层抽象和兼容层说明，而领域词和项目语义仍连续
- 规则：把这组文件归到“本人加 AI”而不是“其他协作者”，输出时与稳定基线分开

- 观察：某组文件位于 provider、meta、depends、topo、adapter 骨架等主链位置，且彼此风格高度一致
- 规则：优先把它们建成“核心协作者簇”，而不是仅视作若干异常文件

## 七、样本不足时的正确做法

当某语言只有零星片段时：

1. 明确标注“不足以提炼语言专属习惯”
2. 只输出跨语言共性
3. 真要继续写代码，则采用该语言保守常规写法补齐
4. 同时提示用户提供更多近邻文件

如果高复杂度样本很少，且疑似 AI 介入，不要把它们当成作者本人稳定上限。

如果某类风格只稳定出现在单一子目录，且疑似来自协作者，不要把它外推为用户全局习惯。

如果多人项目里缺少低污染基线，不要强行完成“用户本人画像”；宁可退回到来源分簇和待确认结论。

## 八、最常见误判

### 误判 1：把“看起来专业”当个人风格

修正：只认复现证据，不认主观印象。

### 误判 2：把单个高手写的基础设施模块当成用户全局风格

修正：看它是否跨模块复现。

### 误判 3：把某语言习惯迁移到另一种语言

修正：每种语言单独建模，只在跨语言层写真正共性。

### 误判 4：为了凑画像而硬写结论

修正：结论不够稳就留空，或降级成弱信号。

### 误判 5：把英语不标准的命名一律当成低水平

修正：先判断那是不是项目术语、社区黑话、拼音专名或作者故意保留的领域词。

### 误判 6：只看类名和目录，不看函数内部

修正：至少抽样阅读若干复杂函数，确认局部变量命名、分支偏好、循环偏好和中间状态组织方式。

### 误判 7：把 AI 写得像作者的代码直接当作者稳定习惯

修正：检查它是否和作者其余样本连续、是否有演进痕迹、是否在相邻文件中也复现。

### 误判 8：把作者偶尔借 AI 完成的高复杂度模块，当成作者本人一直具备的能力上限

修正：输出时增加“能力边界”层，区分稳定能力和疑似 AI 补强能力。

### 误判 9：把未署名协作者的代码误当成用户本人风格

修正：检查风格断裂是否呈现为另一套稳定的人类习惯，并在影响画像时直接向用户确认来源。

### 误判 10：把用户本人通过 AI / Vibe Coding 完成的代码误判成其他协作者

修正：检查它是否仍然延续用户的领域语义、项目锚点和局部习惯，并把“本人加 AI”作为独立来源分类。

### 误判 11：先被最吵的实验文件吸引，结果错过核心协作者

修正：优先沿项目主链和高中心性文件寻找稳定作者簇，再回头看实验和兼容层。
