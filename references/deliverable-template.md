# 输出与交付模板

本文件存放 `pretend-handwritten` 的标准输出模板与最终交付物格式。主 `SKILL.md` 不应重复展开这些长模板，只有在真正进入 Phase 4 / Phase 5 时才读取此文件。

## 二、生成出的模仿 Skill 包结构

```text
generated-style-skill/
├── SKILL.md
├── README.md
├── references/
│   ├── style-profile.md
│   └── representative-snippets.md
└── evals/
    └── evals.json
```

### 模仿 Skill 的 `SKILL.md` 至少包含

1. 适用场景
2. 跨语言共性规则
3. 分语言模仿规则
4. 仿写决策顺序
5. 运行时约束
6. 证据优先级
7. 保守补全策略

其中 `运行时约束` 必须明确：

- 不在下游 Skill 运行时重新做来源归因
- 不依赖完整源码、历史项目名、目录名来识别协作者或 AI 区域
- 无法确认来源时，默认只按近邻文件与高置信度规则续写

## 三、代表性代码片段格式

```markdown
## snippet-python-01
- 语言：Python
- 来源文件：path/to/file.py
- 说明：为什么是代表性片段（命名风格 / 常用小模式 / 错误习惯）
- 片段（3–6 行）：

```python
def compute_total(items):
    total = 0
    for it in items:
        total += it.price
    return total
```
```

## 四、示例：输入 / 输出

### 示例 1：提炼画像并生成模仿 Skill

- 输入（用户）：“下面是我 3 个仓库，请根据这些代码提炼我的编码习惯，并生成一个可直接用于模仿我写码风格的 Skill。”
- 输出（skill）：
  - 跨语言共性（3 条高置信度规则）
  - Python、JavaScript 各自的 5 条高置信度模仿规则
  - 5 个代表性代码片段（每个 3–6 行），标注来源文件路径
  - 一个完整的模仿 Skill 包

### 示例 2：给出样本并要求按风格补代码

- 输入（用户）：“这是目标模块的近邻文件，请按我的风格实现 `process_order()`。”
- 输出（skill）：
  - 极短的风格确认（2–4 条高置信度规则）
  - 目标语言下的实现代码（按用户风格）
  - 在实现后附上“本次使用了哪些风格规则”和“哪些点属于保守补全”
